# 智慧城市边缘视频分析平台 - 架构设计说明书

## 一、项目概述

### 1.1 项目目标
搭建一套边缘端视频分析演示平台，对接实验室模拟路灯摄像设备，实现：
- 设备接入与状态管理
- 视频流拉取与AI推理调度
- 传感器数据采集与监控
- 历史数据可视化查询
- 远程设备控制

### 1.2 技术栈
| 层级 | 技术选型 |
|------|----------|
| 后端框架 | Spring Boot 2.7+ |
| ORM | MyBatis-Plus |
| 数据库 | MySQL 8.0 |
| 缓存 | Redis |
| 视频处理 | FFmpeg |
| 实时通信 | WebSocket |
| AI推理 | YOLOv8 (HTTP接口) |
| 容器化 | Docker |
| 异步编程 | CompletableFuture + 自定义线程池 |

### 1.3 开发规范
- **包命名**: `com.edgevideoanalysis.{模块名}`
- **接口命名**: `IXxxService` (接口), `XxxServiceImpl` (实现)
- **控制器命名**: `XxxController`
- **实体命名**: `Xxx` (对应数据库表 `t_xxx`)
- **DTO/VO**: `XxxDTO` (请求参数), `XxxVO` (响应数据)
- **统一响应**: `Result<T>` 封装所有接口返回
- **异常处理**: 全局 `@ControllerAdvice` 统一捕获
- **日志**: SLF4J + Logback，关键操作记录INFO级别日志

## 三、模块职责与边界

### 3.1 模块总览

| 模块 | 职责 | 边界 |
|------|------|------|
| **common** | 公共配置、异常处理、统一响应、常量 | 不依赖任何业务模块，只被依赖 |
| **device** | 灯杆设备CRUD、设备列表、设备详情 | 不依赖其他业务模块 |
| **sensor** | 传感器数据采集、存储、查询、曲线数据 | 依赖device模块获取灯杆信息 |
| **alarm** | 报警规则设置、报警触发、报警记录查询 | 依赖sensor模块获取传感器数据 |
| **video** | 视频流拉取、帧截取、分块压缩传输 | 依赖ai模块进行推理 |
| **ai** | YOLOv8推理调用、推理记录存储、结果查询 | 独立模块，被video依赖 |
| **websocket** | 设备心跳、状态推送、断线重连 | 依赖device模块更新设备状态 |
| **control** | LED灯远程控制、控制指令下发 | 依赖device模块验证设备 |

### 3.2 各模块详细职责

#### 3.2.1 common 公共模块
**职责**: 提供全局基础设施
- `ThreadPoolConfig`: 全局线程池配置（核心4，最大7，有界队列）
- `RedisConfig`: Redis序列化、连接池配置
- `WebSocketConfig`: WebSocket端点注册
- `MyBatisPlusConfig`: 分页插件、逻辑删除配置
- `GlobalExceptionHandler`: 统一异常捕获与返回
- `Result<T>`: 统一响应结构 `{code, message, data}`

**边界**: 纯工具类，不引入任何业务逻辑

#### 3.2.5 video 视频处理模块 【你已优化】
**职责**: 视频流处理与帧截取
- 从摄像头拉取视频流
- 截取当前帧图片
- 异步处理视频帧（线程池 + CompletableFuture）
- Base64分块压缩传输

**接口**:
| 接口 | 方法 | 路径 | 说明 |
|------|------|------|------|
| 当前帧 | GET | `/api/video/frame/{lampId}` | 获取某灯杆当前帧图片 |
| 视频流 | GET | `/api/video/stream/{lampId}` | 获取视频流地址 |

**核心优化点**:
```java
// 线程池配置
corePoolSize: 4
maxPoolSize: 7
queue: LinkedBlockingQueue(100)
rejectPolicy: 自定义拒绝策略（任务过载时丢弃当前帧、记录日志，避免线程阻塞）

// 自定义拒绝策略说明
// 当线程池队列满时，触发 DiscardOldestPolicy 变体：
// 1. 丢弃当前待处理帧（不阻塞调用线程）
// 2. 记录 WARN 级别日志（便于监控吞吐量瓶颈）
// 3. 返回降级响应（提示"当前处理繁忙，请稍后重试"）
// 效果：单帧耗时从 750ms 降至 190ms，避免线程池满载导致系统雪崩

// 异步处理流程
CompletableFuture.supplyAsync(() -> captureFrame(lampId), videoThreadPool)
    .thenApplyAsync(this::compressFrame, videoThreadPool)
    .thenAcceptAsync(this::sendToClient, videoThreadPool);
```

**边界**: 只负责视频流的拉取和帧处理，AI推理交给ai模块

#### 3.2.7 websocket 实时通信模块 【你已优化】
**职责**: 设备心跳与状态推送
- WebSocket连接管理
- 设备心跳校验（Redis版本号机制）
- 设备状态实时推送
- 断线自动重连

**核心机制**:
```java
// 心跳校验
- 设备连接时生成版本号存入Redis: device:heartbeat:{lampId} -> version
- 定时发送心跳包，版本号递增
- 版本号不一致则判定为断线，触发重连
- 状态变更通过WebSocket推送给前端
```

**Redis使用场景**:
| 场景 | Key格式 | 说明 |
|------|---------|------|
| 心跳版本号 | `device:heartbeat:{lampId}` | 心跳校验，版本号递增校验一致性 |
| 设备在线状态 | `device:online:{lampId}` | 缓存设备在线状态，避免频繁查询MySQL |
| 传感器最新数据 | `sensor:latest:{lampId}` | 缓存高频传感器最新数据，减轻MySQL读取压力 |
| WebSocket会话 | `ws:session:{lampId}` | 存储设备WebSocket会话信息，支持断线重连 |

**边界**: 只负责实时通信和状态同步，不处理设备业务逻辑

### 5.1 分层说明

```
┌─────────────────────────────────────────────────┐
│                  Controller 层                   │
│  职责: 接收HTTP请求，参数校验，调用Service，返回响应  │
│  边界: 不包含业务逻辑，只做请求转发和响应封装         │
└─────────────────────────────────────────────────┘
                        │
                        ▼
┌─────────────────────────────────────────────────┐
│                  Service 层                      │
│  职责: 核心业务逻辑，事务管理，数据组装             │
│  边界: 不直接操作数据库，通过Mapper访问数据         │
└─────────────────────────────────────────────────┘
                        │
                        ▼
┌─────────────────────────────────────────────────┐
│                  Mapper 层                       │
│  职责: 数据库CRUD操作，SQL执行                    │
│  边界: 只做数据访问，不包含业务逻辑                │
└─────────────────────────────────────────────────┘
                        │
                        ▼
┌─────────────────────────────────────────────────┐
│                  Entity 层                       │
│  职责: 数据库表映射，数据载体                      │
│  边界: 纯数据类，不包含业务方法                    │
└─────────────────────────────────────────────────┘
```

### 5.2 DTO/VO 使用规范

| 类型 | 用途 | 示例 |
|------|------|------|
| **DTO** | 接收前端请求参数 | `LampDTO`, `SensorDataDTO`, `AlarmRuleDTO` |
| **VO** | 返回前端响应数据 | `LampVO`, `SensorDataVO`, `InferenceResultVO` |
| **Entity** | 数据库表映射 | `Lamp`, `SensorData`, `InferenceRecord` |

**转换规则**:
- Controller接收DTO → Service处理 → 返回VO
- Entity只在Service和Mapper层使用
- 使用MapStruct或手动转换，禁止Entity直接返回前端

## 七、关键业务流程

### 7.1 视频分析完整流程

```
前端请求当前帧
       │
       ▼
VideoController → VideoStreamService
       │
       ▼
[异步] CompletableFuture 拆分任务
       ├── 任务1: FFmpeg截取当前帧
       ├── 任务2: Base64分块压缩
       └── 任务3: 调用AI推理
              │
              ▼
         AIInferenceAdapter → YOLOv8 HTTP接口
              │
              ▼
         返回: {processedImage, personCount, inferenceResults}
              │
              ▼
         存储推理记录 → 返回前端
```

### 7.2 传感器报警流程

```
设备上报传感器数据
       │
       ▼
SensorController → SensorDataService
       │
       ├── 存储传感器数据
       │
       ▼
AlarmRuleService 查询该灯杆报警规则
       │
       ├── 判断是否超出上下限
       │
       ▼
[触发报警] AlarmRecordService 创建报警记录
       │
       ▼
WebSocket 推送报警信息到前端
```

### 7.3 设备心跳流程

```
设备WebSocket连接
       │
       ▼
DeviceWebSocketHandler 注册连接
       │
       ├── Redis写入版本号: device:heartbeat:{lampId} = 1
       │
       ▼
定时心跳: 设备发送心跳包
       │
       ├── 版本号递增: device:heartbeat:{lampId} = 2
       │
       ▼
[版本号不一致] → 判定断线 → 触发重连
       │
       ▼
状态变更 → WebSocket推送前端更新设备状态
```

## 九、异常处理策略

```java
// 业务异常
public class BusinessException extends RuntimeException {
    private Integer code;
    private String message;
}

// 全局异常处理
@ControllerAdvice
public class GlobalExceptionHandler {
    
    @ExceptionHandler(BusinessException.class)
    public Result<?> handleBusinessException(BusinessException e) {
        return Result.error(e.getCode(), e.getMessage());
    }
    
    @ExceptionHandler(Exception.class)
    public Result<?> handleException(Exception e) {
        log.error("系统异常", e);
        return Result.error("系统异常，请联系管理员");
    }
}
```
